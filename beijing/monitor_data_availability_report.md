# 多机构预报数据时效性监控分析报告

## 一、汇报结论

本次监控结果表明：接口请求成功并不等于目标时次的业务数据已经发布。各预报机构存在明显的数据发布时差，最新起报时次通常需要等待约 9～10 小时才会出现首个有效机构数据；如果要求多机构数据较完整，通常需要等待 16～20 小时以上。

因此，系统不宜在起报时次到达后立即判定数据缺失，应增加数据时效判断、定时重试和预报时次回退机制。

### 2. 监控启动后的典型表现

- BAVI 的 12 时起报数据在约 8.9 小时后出现首个有效机构。
- HAISHEN 的 12 时起报数据在约 9.9 小时后出现首个有效机构。
- 18 时和 00 时起报数据通常在约 9 小时后出现首个有效机构。
- CP-90 的有效数据通常需要约 16 小时。

这说明数据发布并非严格按照起报时间同步落盘，而是存在机构间的异步处理过程。

## 五、问题定位

### 1. 接口状态与业务数据状态不一致

部分请求同时返回：

```json
{
  "http_status": 200,
  "api_code": 200,
  "record_count": 0,
  "available": false
}
```

这说明通信层和接口层均正常，但目标起报时间对应的有效预报记录尚未发布。系统不能只依据 HTTP 状态码或接口业务码判断数据是否可用。

### 2. 外网服务存在偶发网关异常

监控期间出现 18 次 `502 Bad Gateway`，分散在多个机构和多个检查时刻。该问题属于外网链路或代理网关异常，与“接口正常但数据尚未发布”是两类不同问题，应分别统计和处理。

### 3. 日志存在编码问题

监控日志中的“菲律宾”机构名称出现编码异常，导致部分 JSONL 行不严格符合 JSON 格式。该问题不影响当时的接口可用性判断，但会影响日志的后续自动解析、统计和可视化。

## 六、建议的处理策略

### 1. 增加数据可用性判定

同时校验以下条件：

- HTTP 状态码正常；
- 接口业务码正常；
- 返回记录中的 `datetime` 与目标起报时间一致；
- 存在有效的 `fcsthour`、`lat` 和 `lon` 字段。

### 2. 设置分阶段重试窗口

| 时间窗口 | 建议处理方式 |
|---|---|
| 起报后 0～8 小时 | 不判定为异常，等待数据发布 |
| 起报后 9～10 小时 | 开始周期性检查，优先使用已发布机构 |
| 起报后 16 小时 | 尝试获取较完整的多机构数据 |
| 起报后 20 小时以上 | 若仍无数据，触发告警或使用历史时次回退 |

### 3. 增加预报时次回退机制

当最新起报时次尚未形成有效数据时，按照时间倒序尝试上一可用时次，例如：

```text
当前时次 -> 前一时次 -> 前两时次 -> 更早时次
```

回退时必须记录实际使用的起报时间，避免将旧时次数据误标记为最新预报。

### 4. 改进监控程序本身

- 为外网 `502`、连接超时和业务空数据分别设置重试策略。
- 日志统一使用 UTF-8 编码，保证 JSONL 始终可被标准解析器读取。
- 监控进程使用任务计划程序、服务或容器托管，避免隐藏进程意外退出。
- 增加进程存活检测和每日运行状态汇总。

## 七、最终结论

本次问题的核心不是接口完全不可访问，而是多机构预报数据存在发布滞后和到达不同步现象。系统如果在整点后立即请求并要求所有机构同时返回最新时次，就容易出现“接口成功但无有效数据”的结果。

采用“内容校验 + 延迟等待 + 定时重试 + 时次回退”的组合策略后，可以降低空数据误判，避免任务在最新时次上反复请求，并提高多机构预报处理流程的稳定性。
